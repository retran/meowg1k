# Integrations Guide

`meowg1k` is built for automation. Its scriptable, non-interactive design makes it suitable as a component for Git hooks, CI/CD pipelines, and custom scripts. This guide provides practical examples for integrating `meowg1k` into your development workflow.

## 1. Git Hooks

Git hooks are scripts that run automatically at certain points in the Git lifecycle. They can be used for automating tasks like generating commit messages or running pre-commit checks.

### Automatically Generating Commit Messages

This is one of the most powerful integrations. The `prepare-commit-msg` hook runs after you type `git commit` but before the editor opens. We can use it to have `meowg1k` generate the commit message for us.

**Goal:** Automatically populate your commit message with a message generated by `meow commit`.

**Setup:**

1. Create a file named `.git/hooks/prepare-commit-msg`.
2. Paste the following script into the file.
3. Make the script executable: `chmod +x .git/hooks/prepare-commit-msg`.

**Script (`.git/hooks/prepare-commit-msg`):**

```bash
#!/bin/bash
#
# Auto-generates a commit message using 'meow commit'.
# The script only runs for regular commits and allows you to edit the message.
#

COMMIT_MSG_FILE=$1
COMMIT_SOURCE=$2

# Only run for regular commits (not merge, squash, amend, etc.)
if [ -z "$COMMIT_SOURCE" ]; then
    echo "meowg1k: Generating commit message..."
    # Use 'meow commit' and write the output to the commit message file
    meow commit --silent > "$COMMIT_MSG_FILE"
fi
```

Now, when you run `git commit`, your editor will open pre-filled with a generated message, ready for you to review and save.

### Pre-commit Code Analysis

The `pre-commit` hook runs after you type `git commit` but before the `prepare-commit-msg` hook. It's useful for running quick checks on the files you're about to commit.

**Goal:** Run a quick security check on staged Go files before committing.

**Script (`.git/hooks/pre-commit`):**

```bash
#!/bin/bash

echo "meowg1k: Running pre-commit analysis..."

# Get a list of staged Go files
STAGED_GO_FILES=$(git diff --cached --name-only --diff-filter=ACM | grep '\.go$')

if [ -z "$STAGED_GO_FILES" ]; then
    echo "No Go files to analyze."
    exit 0
fi

for file in $STAGED_GO_FILES; do
    echo "Analyzing $file..."
    # Pipe the file content to meowg1k for a quick security check
    # The output is shown but does not block the commit.
    # To block the commit on issues, you would add logic to exit 1 on failure.
    cat "$file" | meow g -u "Quick security check: any obvious issues in this Go code?"
done

exit 0
```

## 2. CI/CD Pipelines

Integrating `meowg1k` into your CI/CD pipeline ensures consistency and automates tasks for your entire team.

### GitHub Actions: Auto-generate PR Descriptions

This workflow automatically generates a PR description when a new pull request is opened.

**Goal:** When a PR is opened, a GitHub Action runs `meow pullrequest` and updates the PR's body with the generated content.

**Setup:** Create a file at `.github/workflows/pr_description.yml`.

**Workflow (`.github/workflows/pr_description.yml`):**

```yaml
name: Auto PR Description

on:
  pull_request:
    types: [opened]

permissions:
  pull-requests: write
  contents: read

jobs:
  generate-pr-description:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4
        with:
          fetch-depth: 0 # Required to compare branches

      - name: Setup Go
        uses: actions/setup-go@v5
        with:
          go-version: "1.25.1"

      - name: Install meowg1k
        run: go install github.com/retran/meowg1k@latest

      - name: Generate PR Description
        id: generate
        run: |
          DESCRIPTION=$(meow pr --base ${{ github.base_ref }} --silent)
          # Use heredoc to properly handle multiline descriptions
          echo "DESCRIPTION<<EOF" >> $GITHUB_OUTPUT
          echo "$DESCRIPTION" >> $GITHUB_OUTPUT
          echo "EOF" >> $GITHUB_OUTPUT
        env:
          # Store your API key as a secret in your repository settings
          MEOW_GEMINI_API_KEY: ${{ secrets.MEOW_API_KEY }}

      - name: Update PR Description
        run: gh pr edit ${{ github.event.pull_request.number }} --body "${{ steps.generate.outputs.DESCRIPTION }}"
        env:
          # The GITHUB_TOKEN is automatically provided by GitHub Actions
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

## 3. Custom Shell Scripts

Because `meowg1k` follows Unix principles (reading from stdin, writing to stdout), it's easy to use in any custom shell script.

### Batch Code Review Script

This script iterates through all Go and Python files in a project, runs a review on each one, and saves the output to a results directory.

**Goal:** Perform a bulk analysis of an entire codebase.

**Script (`run_batch_review.sh`):**

```bash
#!/bin/bash
#
# A script to analyze all Go and Python source files in a directory.
#

set -euo pipefail

TARGET_DIRECTORY=${1:-.} # Default to current directory
OUTPUT_DIR="analysis_results"

mkdir -p "$OUTPUT_DIR"
echo "Starting batch analysis. Results will be saved in: $OUTPUT_DIR"

# Find all .go and .py files, excluding vendor directories
find "$TARGET_DIRECTORY" -type f \( -name "*.go" -o -name "*.py" \) -not -path "*/vendor/*" | while read -r file; do
    echo "Analyzing: $file"

    # Generate a clean output filename
    output_filename=$(echo "$file" | tr '/' '_')

    # Run the analysis and save the output
    cat "$file" | meow g -u "Perform a thorough code review and suggest improvements." --silent > "$OUTPUT_DIR/$output_filename.md"
done

echo "Batch analysis complete!"

```

## 4. RAG-Based Development Workflows

Integrate semantic search and code understanding into your daily development workflow.

### Auto-Index on Branch Switch

Automatically keep your index up-to-date when switching branches.

**Goal:** Ensure the RAG index is always current for the checked-out branch.

**Setup:** Add a `post-checkout` Git hook.

**Script (`.git/hooks/post-checkout`):**

```bash
#!/bin/bash
#
# Auto-index codebase after switching branches
#

PREV_HEAD=$1
NEW_HEAD=$2
BRANCH_CHANGE=$3

# Only run if switching branches (not just checking out files)
if [ "$BRANCH_CHANGE" = "1" ]; then
    echo "meowg1k: Branch changed, updating index..."
    meow index --silent
    echo "meowg1k: Index updated successfully"
fi
```

Make it executable:

```bash
chmod +x .git/hooks/post-checkout
```

### Pre-commit Knowledge Check

Ask the AI if your changes follow existing patterns before committing.

**Script (`.git/hooks/pre-commit`):**

```bash
#!/bin/bash

echo "meowg1k: Checking changes against codebase patterns..."

# Get the diff
DIFF=$(git diff --cached)

# Ask AI about the changes
ANSWER=$(echo "$DIFF" | meow ask "Do these changes follow existing patterns in the codebase? Are there any concerns?" --silent)

echo "$ANSWER"

# Continue with commit
exit 0
```

### Editor Integration: Neovim/Vim

Create a custom command to ask questions about the current file or selection.

**Configuration (`~/.config/nvim/init.lua` or `~/.vimrc`):**

For Neovim (Lua):

```lua
-- Ask a question about the current buffer
vim.api.nvim_create_user_command('MeowAsk', function(opts)
    local question = opts.args
    local bufnr = vim.api.nvim_get_current_buf()
    local lines = vim.api.nvim_buf_get_lines(bufnr, 0, -1, false)
    local content = table.concat(lines, '\n')

    local handle = io.popen('meow ask "' .. question .. '" 2>&1', 'r')
    local result = handle:read('*a')
    handle:close()

    -- Display in a new buffer
    vim.cmd('new')
    local new_bufnr = vim.api.nvim_get_current_buf()
    local result_lines = vim.split(result, '\n')
    vim.api.nvim_buf_set_lines(new_bufnr, 0, -1, false, result_lines)
    vim.bo[new_bufnr].buftype = 'nofile'
    vim.bo[new_bufnr].filetype = 'markdown'
end, { nargs = 1 })

-- Search for code
vim.api.nvim_create_user_command('MeowQuery', function(opts)
    local query = opts.args
    local handle = io.popen('meow query "' .. query .. '" 2>&1', 'r')
    local result = handle:read('*a')
    handle:close()

    vim.cmd('new')
    local bufnr = vim.api.nvim_get_current_buf()
    local lines = vim.split(result, '\n')
    vim.api.nvim_buf_set_lines(bufnr, 0, -1, false, lines)
    vim.bo[bufnr].buftype = 'nofile'
    vim.bo[bufnr].filetype = 'text'
end, { nargs = 1 })
```

For Vim (VimScript):

```vim
" Ask a question about the codebase
command! -nargs=1 MeowAsk call s:MeowAsk(<q-args>)
function! s:MeowAsk(question)
    let result = system('meow ask "' . a:question . '"')
    new
    setlocal buftype=nofile
    setlocal filetype=markdown
    call setline(1, split(result, '\n'))
endfunction

" Search for code
command! -nargs=1 MeowQuery call s:MeowQuery(<q-args>)
function! s:MeowQuery(query)
    let result = system('meow query "' . a:query . '"')
    new
    setlocal buftype=nofile
    call setline(1, split(result, '\n'))
endfunction
```

**Usage in Neovim/Vim:**

```vim
:MeowAsk How does authentication work?
:MeowQuery database connection logic
```

### Slack Bot Integration

Create a Slack bot that answers questions about your codebase.

**Example using a simple webhook:**

```bash
#!/bin/bash
# slack_meow_bot.sh

# Slack webhook URL (create one in your Slack workspace)
WEBHOOK_URL="https://hooks.slack.com/services/YOUR/WEBHOOK/URL"

# Read the question from Slack
QUESTION="$1"

# Get answer from meow
ANSWER=$(meow ask "$QUESTION" --silent)

# Post to Slack
curl -X POST "$WEBHOOK_URL" \
  -H 'Content-Type: application/json' \
  -d "{\"text\":\"Q: $QUESTION\n\nA: $ANSWER\"}"
```

**More robust solution with a web server:**

```python
from flask import Flask, request, jsonify
import subprocess

app = Flask(__name__)

@app.route('/slack/ask', methods=['POST'])
def slack_ask():
    data = request.form
    question = data.get('text', '')

    if not question:
        return jsonify({"text": "Please provide a question!"}), 200

    # Run meow ask
    result = subprocess.run(
        ['meow', 'ask', question, '--silent'],
        capture_output=True,
        text=True
    )

    answer = result.stdout if result.returncode == 0 else "Error getting answer"

    return jsonify({
        "response_type": "in_channel",
        "text": f"Q: {question}\n\nA: {answer}"
    }), 200

if __name__ == '__main__':
    app.run(port=3000)
```

### CI/CD: Documentation Generation

Automatically generate documentation based on code changes.

**GitHub Actions (`.github/workflows/auto_docs.yml`):**

```yaml
name: Auto Documentation

on:
  push:
    branches: [main, dev]

jobs:
  generate-docs:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0 # Need full history for index

      - name: Install meowg1k
        run: |
          curl -L https://github.com/retran/meowg1k/releases/latest/download/meow_linux_amd64.tar.gz | tar xz
          sudo mv meow /usr/local/bin/

      - name: Index codebase
        env:
          MEOW_GEMINI_API_KEY: ${{ secrets.MEOW_GEMINI_API_KEY }}
        run: meow index

      - name: Generate architecture documentation
        env:
          MEOW_GEMINI_API_KEY: ${{ secrets.MEOW_GEMINI_API_KEY }}
        run: |
          meow ask "Describe the overall architecture of this project" > docs/ARCHITECTURE.md
          meow ask "What are the main components and their responsibilities?" >> docs/ARCHITECTURE.md

      - name: Commit documentation
        run: |
          git config --local user.email "action@github.com"
          git config --local user.name "GitHub Action"
          git add docs/
          git diff --quiet && git diff --staged --quiet || git commit -m "docs: auto-generate architecture documentation"
          git push
```
